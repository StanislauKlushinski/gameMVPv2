# Оффлайн прогресс (AFK расчёт)

Оффлайн прогресс (AFK расчёт)

### 11.1. Определение

Оффлайн прогресс — система симуляции производства за время отсутствия игрока.

### 11.2. Базовая модель

```text
offline_progress = simulate(game_state, offline_time)
```

### 11.3. Ограничения симуляции

#### 11.3.1. Максимальное время

```text
max_offline_time = X часов (например 8-12)
```

**Если больше:**

```text
offline_time = max_offline_time
```

### 11.4. Упрощённая симуляция

**Оффлайн режим НЕ должен:**
- выполнять полный tick симуляции
- рассчитывать каждую секунду

**Вместо этого:**

```text
result = production_rate * effective_time
```

### 11.5. Эффективное время

```text
effective_time = offline_time * efficiency_factor
```

**Где:**
- early game: 0.3 - 0.6
- mid game: 0.6 - 0.9
- late game: 0.8 - 1.0

### 11.6. Ограничения систем

**Во время оффлайна:**

#### 11.6.1. Логистика

упрощается

считается мгновенной или усреднённой

#### 11.6.2. Производственные цепочки

только итоговый результат, без поцикловой симуляции

#### 11.6.3. Узкие места

bottlenecks учитываются через коэффициенты

### 11.7. Формула производства

```text
output = sum(all_buildings_production) * effective_time
```

### 11.8. Ограничение перепроизводства

output <= storage_limit

### 11.9. Переполнение склада

**Если превышение:**
- излишек теряетсяили
- конвертируется в бонус (опционально)

### 11.10. Влияние технологий

offline_bonus += tech_modifier

### 11.11. Влияние престижа

offline_efficiency += prestige_bonus

### 11.12. Состояние зданий

**При оффлайне:**
- ACTIVE → считается
- IDLE → 0
- BLOCKED → 0

### 11.13. Проблема "идеального оффлайна"

**Нельзя допускать:**

```text
offline = better than active play
```

**Решение:**
- штраф эффективности
- ограничение времени
- упрощение логики

### 11.14. UI требования

**Игрок должен видеть:**
- сколько времени прошло
- сколько ресурсов получено
- какие здания работали
- какие простаивали

### 11.15. Формула результата

```text
final_resources =production_rate* offline_time* efficiency_multiplier* caps
```

### 11.16. Edge cases

#### 11.16.1. Нулевое производство

ничего не начисляется

#### 11.16.2. Переполненный склад

лишнее теряется

#### 11.16.3. Отсутствие ресурсов

производство считается остановленным

### 11.17. Ошибки проектирования

Слишком точная симуляция (дорого)

Игнорирование bottlenecks

Отсутствие капа времени

Перекос в пользу оффлайна

### 11.18. Минимальный набор (definition of done)

**Система реализована, если:**
- есть расчёт оффлайн прогресса
- есть лимит времени
- есть упрощение симуляции
- есть награда игроку
- нет превышения эффективности активной игры
