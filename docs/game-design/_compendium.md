## 1. Общая концепция

Игра - 2D AFK factory-симулятор с несколькими параллельными эпохами, где игрок строит производственные цепочки, оптимизирует потоки ресурсов и постепенно открывает новые технологические слои.

## 2. Игровой цикл

**Основной цикл:**
- Добыча ресурсов
- Переработка ресурсов в промежуточные продукты
- Автоматическое распределение ресурсов между зданиями
- Улучшение производственных цепочек
- Открытие новых технологий и эпох
- Ускорение прогресса через глобальные улучшения
- Престиж -> сброс прогресса с постоянными бонусами

## 3. Мир и структура

### 3.1. Эпохи

Каждая эпоха - отдельная производственная зона.

**Примеры:**
- Каменный век
- Античность
- Индустриальная эпоха
- Космическая эпоха

**Особенности:**
- все эпохи существуют одновременно
- игрок переключается между ними
- каждая эпоха производит свои ресурсы и здания

### 3.2. Связь эпох

ресурсы могут использоваться в других эпохах

эффекты из поздних эпох могут усиливать ранние

ранние ресурсы могут оставаться актуальными всю игру

## 4. Ресурсная система

### 4.1. Типы ресурсов

Базовые

добываются напрямую (дерево, камень, руда)

Промежуточные

создаются в зданиях (доски, инструменты, детали)

Продвинутые

используются в поздних эпохах (сплавы, энергия, компоненты)

## 5. Здания

### 5.1. Типы зданий

## 1. Добывающие

создают базовые ресурсы

работают постоянно

## 2. Производственные

перерабатывают ресурсы в новые

требуют входные материалы

## 3. Жилые

увеличивают количество рабочих

## 4. Логистические

увеличивают скорость/объём передачи ресурсов

## 5. Технологические

открывают новые рецепты и эпохи

### 5.2. Поведение зданий

**Каждое здание:**
- потребляет ресурсы (если нужно)
- производит результат по таймеру
- имеет скорость работы
- может улучшаться

## 6. Рабочие

Рабочие - не логистический элемент, а модификатор.

**Роль:**
- увеличивают эффективность зданий
- дают бонус к скорости производства

**Важно:**
- не участвуют в переносе ресурсов
- не ломают систему логистики
- служат визуальной и числовой надстройкой

## 7. Логистика

### 7.1. Базовая модель

общий пул ресурсов внутри эпохи

здания автоматически берут и кладут ресурсы

## 8. Эпохи как система прогрессии

**Каждая эпоха:**
- открывает новые здания
- добавляет новые ресурсы
- усложняет цепочки производства
- Ключевая идея

**Игрок не "проходит" эпоху, а:**
- добавляет её как новый слой производства
- старые эпохи остаются активными и полезными

## 9. Технологии

**Дерево технологий:**
- открывает здания
- открывает ресурсы
- улучшает параметры систем

**Особенность:**

технологии могут усиливать другие эпохи

## 10. Престиж

**Механика:**

полное обнуление прогресса

**Награда:**
- глобальные множители скорости
- ускорение производства
- усиление кросс-эпохных связей
- возможно открытие новых систем

## 11. Оффлайн прогресс

работает частично

ограниченное время симуляции

сниженная эффективность

упрощённые расчёты (без сложных цепочек)

## 12. Основная цель

**Финальная цель:**

запуск ракеты

**Это:**
- требует ресурсов всех эпох
- требует полной оптимизации цепочек
- является точкой завершения цикла

**После:**

престиж и новый цикл развития

## 01. Игровой цикл (core loop)

Игровой цикл (core loop)

### 1.1. Определение

Игровой цикл — непрерывный процесс преобразования ресурсов с целью увеличения скорости производства и открытия нового контента.

Ресурсы -> Производство -> Улучшения -> Ускорение -> Новые возможности -> Повтор

### 1.2. Состояния цикла

**Цикл разбивается на повторяющиеся состояния:**
- Генерация ресурсов
- Потребление ресурсов
- Преобразование (крафт)
- Накопление
- Инвестирование (строительство/апгрейды)
- Ускорение
- Разблокировка нового контента

### 1.3. Детализация этапов

#### 1.3.1. Генерация ресурсов

**Источник:**
- добывающие здания
- производственные здания

**Формула:**

```text
production_rate = base_rate * building_multiplier * worker_multiplier * global_multiplier
```

**Где:**
- base_rate — базовая скорость здания
- building_multiplier — уровень здания
- worker_multiplier — влияние рабочих
- global_multiplier — технологии, престиж

#### 1.3.2. Потребление ресурсов

**Каждое производственное здание:**
- Условие работы:
- если (все входные ресурсы >= требуемых) -> работаетиначе -> простаивает

**Потребление:**

input_resources -= recipe_cost * delta_time

#### 1.3.3. Преобразование (крафт)

**Производство по рецепту:**

output += recipe_output * efficiency * delta_time

**Где:**
- efficiency зависит от:
- рабочих
- апгрейдов
- глобальных бонусов

#### 1.3.4. Накопление

**Все ресурсы складываются в хранилище эпохи:**
- Ограничение:
- resource <= storage_limit

**Если превышен лимит:**

```text
production = 0 (или уменьшается)
```

#### 1.3.5. Инвестирование

**Игрок тратит ресурсы на:**
- строительство зданий
- улучшение зданий
- технологии

**Формат стоимости:**

```text
cost = base_cost * growth_factor^level
```

#### 1.3.6. Ускорение

**Источники ускорения:**
- рабочие
- технологии
- глобальные бонусы
- престиж

**Общая формула:**

```text
total_multiplier = product(всех множителей)
```

#### 1.3.7. Разблокировки

**Условия:**
- накопление ресурсов
- выполнение порога
- покупка технологии

**Результат:**
- новые здания
- новые ресурсы
- новая эпоха

### 1.4. Временная модель

#### 1.4.1. Тик системы

**Игра работает по тиковой модели:**

```text
tick_rate = 0.1 - 1 сек
```

**На каждом тике:**
- обновляются все здания
- пересчитываются ресурсы

#### 1.4.2. Delta time

**Все расчёты через:**

delta_time

**Это нужно для:**
- оффлайн прогресса
- стабильности симуляции

### 1.5. Приоритеты обработки

**Порядок обновления на тике:**
- Проверка доступности ресурсов
- Потребление ресурсов
- Производство ресурсов
- Применение ограничений (лимиты)
- Обновление UI

### 1.6. Поведение при дефиците

**Если не хватает ресурсов:**

здание останавливается полностью

### 1.7. Поведение при избытке

**Если склад заполнен:**

стоп производства

### 1.8. Цикл игрока (поведение)

**Игрок постоянно делает:**
- анализ узких мест (bottlenecks)
- увеличение производства нужного ресурса
- перераспределение рабочих
- покупка апгрейдов
- открытие нового контента

### 1.9. Узкие места (ключевая механика)

**Игра строится на:**

```text
bottleneck = ресурс с минимальной скоростью производства
```

**Игрок:**
- находит bottleneck
- устраняет его
- создаёт новый bottleneck

### 1.10. Связь с другими системами

**Игровой цикл напрямую связан с:**
- Ресурсами -> что производится
- Зданиями -> как производится
- Рабочими -> скорость
- Логистикой -> доступность
- Технологиями -> разблокировки
- Престижем -> ускорение цикла

### 1.11. Требования к реализации

Все расчёты должны быть детерминированы

Система должна поддерживать большие числа (idle scaling)

Все множители должны быть прозрачны игроку

Любой простой должен быть объясним

**Пересчёт должен работать для:**
- онлайн
- оффлайн

### 1.12. Минимальная проверка (definition of done)

**Игровой цикл считается реализованным, если:**
- ресурсы генерируются автоматически
- здания потребляют и производят
- есть лимиты
- можно строить и улучшать
- есть хотя бы 1 bottleneck
- игрок может ускорять производство

## 02. Ресурсы (типы и уровни)

Ресурсы (типы и уровни)

### 2.1. Определение

**Ресурс — числовая сущность, используемая для:**
- производства
- строительства
- прогрессии

### 2.2. Классификация ресурсов

#### 2.2.1. По уровню

## 1. Базовые

добываются напрямую

не требуют входных ресурсов

**Примеры:**
- дерево
- камень
- руда

## 2. Промежуточные

создаются из базовых или других промежуточных

**Примеры:**
- доски
- кирпич
- инструменты

## 3. Продвинутые

используются в поздних цепочках

**Примеры:**
- механизмы
- электроника
- топливо

### 2.3. Формальная модель ресурса

**Каждый ресурс описывается структурой:**

```text
Resource {id: stringname: stringtype: enum (base, intermediate, advanced)epoch: referencecurrent_amount: floatstorage_limit: float}
```

### 2.4. Хранение ресурсов

#### 2.4.2. Ограничение хранения

amount <= storage_limit

**Если превышение:**

```text
production = 0
```

#### 2.4.1. Локальное хранилище эпохи

**Каждая эпоха имеет свой пул:**

epoch_resources[resource_id] = amount

### 2.5. Генерация ресурсов

**Происходит через здания:**

```text
delta_resource = production_rate * delta_time
```

**Добавление:**
- resource += delta_resource
- С учётом лимита.

### 2.6. Потребление ресурсов

if resource >= required:resource -= requiredelse:stop_production

### 2.7. Рецепты

Ресурсы преобразуются через рецепты.

**Структура:**

```text
Recipe {input: {resource_id: amount}output: {resource_id: amount}time: float}
```

### 2.8. Скорость производства ресурса

```text
output_per_sec = output_amount / recipe_time * total_multiplier
```

### 2.9. Баланс ресурсов

#### 2.9.2. Избыточный ресурс

**Если:**

production >> consumption

**то ресурс:**
- перестаёт быть значимым
- должен использоваться дальше

#### 2.9.1. Узкое место

```text
bottleneck = min(production_rate(resource_i))
```

### 2.10. Связь ресурсов между эпохами

#### 2.10.2. Ослабление эффекта

```text
effective_value = base_value * cross_epoch_modifier
```

**где:**

```text
cross_epoch_modifier = 0.1 - 0.3
```

#### 2.10.1. Правило доступа

**Ресурс может:**
- использоваться внутри своей эпохи
- использоваться в других эпохах (если разрешено)

### 2.11. Масштабирование чисел

**Idle требует роста:**

```text
value = base * (growth_factor ^ level)
```

**Требования:**

поддержка больших чисел (double или BigNumber)

**отображение:**

K, M, B, T и т.д.

### 2.12. Приоритеты потребления

**Если ресурс используется в нескольких местах:**
- Варианты:
- равномерно
- по приоритету
- по порядку

**Рекомендуется:**

```text
priority_system = true
```

(игрок может задавать приоритет зданий

### 2.13. Дефицит ресурса

**Если ресурс заканчивается:**
- здание полностью останавливается
- частичная работа не допускается (упрощение логики)

### 2.14. UI требования

**Игрок должен видеть:**
- текущее количество
- скорость (+X/сек)
- лимит хранения
- потребление (-Y/сек)
- дефицит (индикатор)

### 2.15. Ошибки, которых нужно избегать

Ресурс без использования

Бесконечный избыток без лимита

Непрозрачное потребление

Слишком сложные рецепты на старте

### 2.16. Минимальный набор (definition of done)

**Система ресурсов считается реализованной, если:**
- есть минимум 3 ресурса
- есть хотя бы 1 цепочка (A -> B -> C)
- есть лимит хранения
- есть производство и потребление
- есть отображение скорости

## 03. Потоки ресурсов (производство - склад - потребление)

Потоки ресурсов (производство -> склад -> потребление)

### 3.1. Определение

Поток ресурсов — система автоматического перемещения ресурсов от производителей к потребителям внутри эпохи (и между эпохами при наличии разрешения).

### 3.2. Базовая модель

**Каждый тик:**

## 1. здания производят ресурсы2. ресурсы попадают в пул эпохи3. здания запрашивают ресурсы из пула4. при наличии ресурсов запускается производство

### 3.3. Архитектура потоков

#### 3.3.1. Глобальный пул эпохи

```text
EpochResourcePool {resource_id -> amount}
```

#### 3.3.2. Очередь потребления

**Все здания-потребители формируют очередь:**

```text
ConsumersQueue = sorted(buildings by priority)
```

### 3.4. Алгоритм распределения

**На каждом тике:**

for building in ConsumersQueue:if pool >= required:allocate resourceselse:skip building

### 3.5. Приоритеты

**Каждое здание имеет:**

priority: int (0..N)

**Правила:**
- выше значение -> раньше получает ресурсы
- одинаковый приоритет -> равномерное распределение

### 3.6. Режимы распределения

#### 3.6.1. Жёсткий

если не хватает ресурсов -> здание не работает

### 3.7. Буферизация

#### 3.7.1. Внутренний буфер здания

**Каждое здание имеет локальный запас:**

internal_storage[resource] = amount

**Назначение:**
- сглаживание скачков
- уменьшение "дёргания" производства

#### 3.7.2. Логика буфера

если internal_storage < threshold:взять ресурсы из пула

### 3.8. Ограничения потока

#### 3.8.1. Скорость передачи

```text
transfer_rate = base_rate * logistics_multiplier
```

#### 3.8.2. Лимит передачи

max_transfer_per_tick

### 3.9. Узкие места (bottlenecks)

**Типы:**
- ресурсный не хватает ресурса
- логистический ресурс есть, но не успевает доставляться
- приоритетный ресурс "съедают" другие здания

### 3.10. Кросс-эпохные потоки

#### 3.10.1. Доступ

if cross_epoch_unlocked:allow_transfer

#### 3.10.2. Ограничение

cross_epoch_transfer *= penalty_multiplier

#### 3.10.3. Буфер эпох

**Рекомендуется:**

InterEpochBuffer

**Назначение:**
- контролировать поток между эпохами
- избегать мгновенного "перелива"

### 3.11. Поведение при дефиците

если ресурса нет:здание = idle

**Индикаторы:**
- "нет ресурса"
- "низкий приоритет"

### 3.12. Поведение при избытке

если storage заполнен:stop input

### 3.13. Визуализация (важно)

**Игрок должен видеть:**
- поток (+X/сек)
- потребление (-Y/сек)
- источник ресурса
- узкое место

### 3.14. Требования к производительности

O(N) или O(N log N) на тик

без поиска путей (pathfinding отсутствует)

без физической симуляции

### 3.15. Ошибки, которых нужно избегать

Ручное соединение всех зданий

Непредсказуемое распределение

"Магическое исчезновение" ресурсов

Скрытые приоритеты

### 3.16. Минимальный функционал (definition of done)

**Система потоков считается реализованной, если:**
- ресурсы автоматически распределяются
- есть приоритеты зданий
- есть хотя бы 1 bottleneck
- здания могут простаивать
- игрок понимает, почему нет ресурса

## 04. Здания (типы и функции)

Здания (типы и функции)

### 4.1. Определение

**Здание — основная производственная единица, которая:**
- генерирует ресурсы
- преобразует ресурсы
- влияет на систему (логистика, лимиты, прогрессия)

### 4.2. Базовая структура


## 05. Поведение зданий (таймеры, апгрейды)

Поведение зданий (таймеры, апгрейды)

### 5.1. Определение

**Поведение здания — это логика, по которой оно:**
- потребляет ресурсы
- обрабатывает время
- производит результат
- реагирует на ограничения

### 5.2. Модель работы (основа)

**Каждое здание работает по циклу:**

## 1. Проверка условий2. Потребление ресурсов3. Обработка времени4. Производство результата5. Переход к следующему циклу

### 5.3. Типы производственного поведения

#### 5.3.1. Непрерывное (continuous)

**Ресурсы производятся плавно:**

output += rate * delta_time

**Используется для:**
- добычи
- простых производств

#### 5.3.2. Дискретное (tick/cycle)

**Производство по завершению цикла:**

если timer >= cycle_time:произвести outputtimer = 0

**Используется для:**
- сложных рецептов
- крафта

### 5.4. Таймер здания

timer += delta_time * speed_multiplier

### 5.5. Скорость работы

```text
speed_multiplier = worker_bonus *building_level_bonus *global_bonus
```

### 5.6. Потребление ресурсов

#### 5.6.1. Для дискретной модели

при старте цикла:списать input ресурсы

#### 5.6.2. Для непрерывной модели

input -= rate * delta_time

### 5.7. Проверка запуска

**Здание может начать цикл если:**

input >= requiredAND output_storage_not_full

### 5.8. Буфер входа

**Каждое здание:**

internal_storage[resource]

**Используется для:**
- хранения входных ресурсов
- предотвращения "дёргания"

### 5.9. Буфер выхода

output_buffer

**Если:**

output_buffer >= limit:stop production

### 5.10. Состояния здания

```text
ACTIVEWAITING_FOR_INPUTWAITING_FOR_OUTPUT_SPACEDISABLED
```

### 5.11. Прерывание цикла

**Если ресурсы закончились:**
- Вариант (рекомендуемый):
- цикл не начинается

**Но если уже начался:**
- цикл завершается
- (ресурсы уже списаны)

### 5.12. Параллельные циклы

**Если здание может:**

```text
parallel_jobs = N
```

**Тогда:**

timer[i] для каждого слота

### 5.14. Эффект рабочих

```text
worker_bonus = 1 + workers * factor
```

**Ограничение:**

max_workers_per_building

#### 5.13.2. Производство

output *= (1 + level_multiplier)

### 5.13. Эффект уровня

#### 5.13.1. Скорость

cycle_time /= (1 + level_bonus)

### 5.15. Апгрейды здания

**Типы:**
- Скорость
- Объём производства
- Эффективность (меньше затрат)
- Буфер

### 5.16. Формула апгрейда

```text
new_value = base_value * (upgrade_multiplier ^ level)
```

### 5.17. Включение/выключение

```text
is_active = true/false
```

**Если выключено:**
- таймер не идёт
- ресурсы не потребляются

### 5.18. Поведение при переполнении

если нет места:pause

### 5.19. Поведение при оффлайн прогрессе

**Упрощённая модель:**

```text
cycles = floor(offline_time / cycle_time)
```

```text
total_output = cycles * output
```

### 5.20. Приоритет обновления

**Порядок:**

## 1. Проверка условий2. Запуск новых циклов3. Обновление таймеров4. Завершение циклов

### 5.21. Требования к реализации

Без накопления ошибок таймера

Поддержка больших скоростей

Стабильность при delta_time скачках

Детерминированность

### 5.22. Ошибки проектирования

Частичное потребление (сложная логика)

Зависимость от FPS

"Застревание" таймера

Непрозрачное состояние

### 5.23. Минимальный набор (definition of done)

**Система считается реализованной, если:**
- есть цикл производства
- есть таймер
- есть вход/выход ресурсов
- здание может останавливаться
- есть влияние апгрейдов
- есть влияние рабочих

## 06. Рабочие (модификаторы эффективности)

Рабочие (модификаторы эффективности)

### 6.1. Определение

Рабочие — ограниченный ресурс, который назначается на здания и увеличивает их эффективность.

Рабочие не участвуют в логистике, а только модифицируют производство.

### 6.2. Базовая модель

```text
Workers {total: intavailable: int}
```

### 6.3. Источники рабочих

**Рабочие создаются через:**
- жилые здания
- total_workers += housing_capacity

### 6.4. Назначение рабочих

assigned_workers[building] <= worker_slots

### 6.5. Освобождение рабочих

```text
available = total - sum(assigned_workers)
```

### 6.6. Ограничения

#### 6.6.1. Слоты здания

```text
worker_slots = base_slots + upgrades
```

#### 6.6.2. Глобальный лимит

assigned_workers_total <= total_workers

### 6.7. Эффект рабочих

#### 6.7.1. Базовая формула

```text
worker_bonus = 1 + workers_assigned * factor
```

#### 6.7.2. Альтернативная (с diminishing returns)

```text
worker_bonus = 1 + (workers_assigned ^ 0.7) * factor
```

### 6.8. Типы бонусов

**Рабочие могут влиять на:**
- Скорость - cycle_time /= worker_bonus
- Производство - output *= worker_bonus
- Эффективность - input_cost /= worker_bonus
- (использовать 1 основной тип, чтобы не усложнять)

### 6.9. Переназначение

**Игрок может:**

```text
reassign(workers, building_A -> building_B)
```

**Особенности:**
- мгновенно
- без штрафа (в базовой версии)

### 6.10. Приоритет назначения (опционально)

**Можно добавить авто-распределение:**

```text
auto_assign = true
```

**Алгоритм:**
- сначала здания с высоким приоритетом
- затем остальные

### 6.11. Влияние эпох

**Рабочие могут быть:**
- Вариант 1:
- глобальные (общие на все эпохи)

**Вариант 2:**

локальные (по эпохам)

### 6.12. Кросс-эпохные эффекты

worker_bonus *= cross_epoch_modifier

(если рабочие усиливают другую эпоху)

### 6.13. Визуализация

**Рабочие:**
- отображаются как анимация у здания
- не влияют на логику перемещения

### 6.14. Состояния

IDLE (свободен)ASSIGNED (работает)

### 6.15. Балансные параметры

worker_factormax_workers_per_buildinghousing_capacity_per_building

### 6.16. Ограничения системы

Рабочие не должны быть обязательны для старта здания

Не должны полностью ломать баланс

Не должны заменять апгрейды

### 6.17. Ошибки проектирования

Слишком сильные бонусы

Обязательное наличие рабочих

Сложная система типов рабочих

Зависимость логики от их "перемещения"

### 6.18. Минимальный набор (definition of done)

**Система реализована, если:**
- есть лимит рабочих
- есть назначение на здания
- есть влияние на производство
- есть UI отображение
- есть возможность перераспределения

## 07. Логистика (автоматическое распределение)

Логистика (автоматическое распределение)

### 7.1. Определение

**Логистика — система управления доступностью ресурсов, включающая:**
- скорость передачи
- ограничения перемещения
- буферизацию
- приоритеты распределения

**Отличие от потоков:**
- потоки = логика "кто кому"
- логистика = ограничения "как быстро и сколько"

### 7.2. Уровни логистики

#### 7.2.1. Внутри эпохи (основной)

ресурсы перемещаются через общий пул

без физического пути

#### 7.2.2. Между эпохами

ограниченный канал передачи

отдельные коэффициенты

### 7.3. Основные параметры

```text
Logistics {transfer_rate: floattransfer_limit: floatlatency: float}
```

### 7.4. Скорость передачи

```text
actual_transfer = min(request, transfer_rate * delta_time)
```

### 7.5. Лимит передачи

transfer_per_tick <= transfer_limit

### 7.6. Задержка

```text
delivery_time = latency
```

### 7.7. Логистические здания

**Влияют на:**
- скорость передачи
- лимиты
- буферы

#### 7.7.1. Пример эффектов

transfer_rate *= (1 + bonus)transfer_limit += value

### 7.8. Буферизация

#### 7.8.1. Уровни буфера

Глобальный (эпоха)

Локальный (здание)

Межэпохный

#### 7.8.2. Поведение

если глобальный пул пуст:использовать внутренний буфер

### 7.9. Узкие места логистики

#### 7.9.2. Определение

if demand > transfer_rate:bottleneck = logistics

#### 7.9.1. Типы

Недостаточная скорость

Малый лимит

Перегрузка системы

### 7.10. Приоритеты

**Связаны с потоками:**

high_priority_buildings получают ресурсы первыми

**Но логистика ограничивает:**

даже при высоком приоритете ресурс может не дойти

### 7.11. Межэпохная логистика

#### 7.11.1. Канал передачи

inter_epoch_transfer_rate

#### 7.11.2. Штраф

transfer *= penalty

#### 7.11.3. Буфер

inter_epoch_buffer

### 7.12. Масштабирование

**Логистика должна расти:**

```text
transfer_rate = base * (scale ^ level)
```

### 7.13. Поведение при перегрузке

**Если:**

requests > capacity

**то:**
- часть зданий не получает ресурсы
- возникает логистический bottleneck

### 7.14. UI требования

**Игрок должен видеть:**
- скорость передачи
- лимит
- загрузку (использование %)
- наличие bottleneck

### 7.15. Упрощение

без задержек

без сложных маршрутов

без pathfinding

### 7.16. Ошибки проектирования

Логистика не влияет на игру

Слишком сложная система

Невидимые ограничения

Отсутствие bottleneck

### 7.17. Минимальный набор (definition of done)

**Система реализована, если:**
- есть ограничение скорости передачи
- есть логистический bottleneck
- логистические здания влияют на поток
- игрок может улучшать логистику

## 08. Технологии (дерево развития)

Технологии (дерево развития)

### 8.1. Определение

**Технологии — система разблокировок, которая управляет доступом к:**
- зданиям
- ресурсам
- эпохам
- улучшениям

### 8.2. Структура технологии

```text
Technology {id: stringname: stringepoch: referencecost: map<resource, float>unlocks: list<Unlock>prerequisites: list<Technology>is_unlocked: bool}
```

### 8.3. Типы эффектов (Unlock)

```text
Unlock {type: enumvalue: any}
```

#### 8.3.1. Виды unlock-эффектов

Новое здание

```text
unlock_building(building_id)
```

Новый ресурс

```text
unlock_resource(resource_id)
```

Новая эпоха

```text
unlock_epoch(epoch_id)
```

Глобальный бонус

global_multiplier += value

Улучшение здания

```text
modify_building(building_id, parameter)
```

Логистика

increase_transfer_rate

### 8.4. Дерево технологий

#### 8.4.1. Структура

граф (не обязательно строго дерево)

узлы связаны через prerequisites

#### 8.4.2. Условия открытия

```text
can_unlock = (все prerequisites выполнены)AND (ресурсы >= cost)
```

### 8.5. Покупка технологии

if can_unlock:deduct costapply unlocksis_unlocked = true

### 8.6. Стоимость технологий

#### 8.6.1. Формула

```text
cost = base_cost * (scale ^ tier)
```

#### 8.6.2. Типы стоимости

ресурсы текущей эпохи

ресурсы предыдущих эпох

редкие ресурсы

### 8.7. Тиры технологий

**Каждая эпоха делится на уровни:**

```text
tier = 1..N
```

**Назначение:**
- контроль прогрессии
- постепенное усложнение

### 8.8. Связь с эпохами

#### 8.8.1. Внутри эпохи

открывают здания

открывают рецепты

#### 8.8.2. Между эпохами

открывают новую эпоху

усиливают другие эпохи

### 8.9. Кросс-эпохные технологии

apply bonus to other epoch with modifier

**Пример:**

+20% к добыче в каменном веке из индустриальной эпохи

### 8.10. Глобальные технологии

**Не привязаны к эпохе:**
- ускорение производства
- увеличение лимитов
- улучшение логистики

### 8.11. Ограничения

#### 8.11.1. Линейность

**Не допускать:**

полного "разветвления без смысла"

**Рекомендуется:**

2-3 варианта выбора максимум

#### 8.11.2. Блокировки

soft lock (рекомендуется)

(игрок может открыть всё, но не сразу)

### 8.12. UI требования

**Игрок должен видеть:**
- стоимость
- эффект
- зависимости
- статус (доступно / недоступно / куплено)

### 8.13. Баланс

**Технологии должны:**
- Открывать новый геймплей
- Усиливать существующий
- Не давать слишком резкий скачок

### 8.14. Ошибки проектирования

Бесполезные технологии

Слишком линейное дерево

Слишком сложные зависимости

Отсутствие выбора

### 8.15. Минимальный набор (definition of done)

**Система реализована, если:**
- есть дерево технологий
- есть зависимости
- есть разблокировки зданий
- есть разблокировки ресурсов
- есть хотя бы 1 глобальный бонус

## 09. Прогрессия (разблокировки и масштабирование)

Прогрессия (разблокировки и масштабирование)

### 9.1. Определение

**Прогрессия — система, определяющая:**
- темп развития игрока
- порядок открытия контента
- рост сложности и масштабов

### 9.2. Уровни прогрессии

#### 9.2.1. Локальная (внутри эпохи)

**Игрок:**
- открывает здания
- строит цепочки
- увеличивает производство

#### 9.2.2. Межэпохная

**Игрок:**
- открывает новые эпохи
- добавляет новые слои производства

#### 9.2.3. Мета-прогрессия

**Игрок:**
- делает престиж
- ускоряет будущие циклы

### 9.3. Основная модель роста

**Прогрессия основана на:**

production -> reinvestment -> higher production

### 9.4. Кривые роста

#### 9.4.1. Производство

```text
production = base * (scale ^ time)
```

#### 9.4.2. Стоимость

```text
cost = base * (scale ^ level)
```

#### 9.4.3. Баланс

**Условие:**
- growth_production > growth_cost
- иначе игра "умирает".

### 9.5. Разблокировки

**Типы:**
- Здания
- Ресурсы
- Эпохи
- Улучшения

### 9.6. Пороговые условия

**Разблокировка происходит при:**
- resource >= threshold
- или
- technology_unlocked

### 9.7. Темп игры

#### 9.7.1. Фазы

Быстрый старт

Замедление

Ускорение через апгрейды

Плато

Престиж

#### 9.7.2. Цель

**Игрок должен:**
- регулярно получать новые возможности
- не застревать надолго

### 9.8. Узкие места как двигатель

**Прогресс строится через:**

solve(bottleneck) -> create_new_bottleneck

### 9.9. Введение новых механик

**Новые системы вводятся:**
- через технологии
- через эпохи

**Правило:**

не вводить несколько сложных систем одновременно

### 9.10. Масштабирование сложности

#### 9.10.1. По эпохам

больше ресурсов

сложнее рецепты

## 11. Оффлайн прогресс (AFK расчёт)

Оффлайн прогресс (AFK расчёт)

### 11.1. Определение

Оффлайн прогресс — система симуляции производства за время отсутствия игрока.

### 11.2. Базовая модель

```text
offline_progress = simulate(game_state, offline_time)
```

### 11.3. Ограничения симуляции

#### 11.3.1. Максимальное время

```text
max_offline_time = X часов (например 8-12)
```

**Если больше:**

```text
offline_time = max_offline_time
```

### 11.4. Упрощённая симуляция

**Оффлайн режим НЕ должен:**
- выполнять полный tick симуляции
- рассчитывать каждую секунду

**Вместо этого:**

```text
result = production_rate * effective_time
```

### 11.5. Эффективное время

```text
effective_time = offline_time * efficiency_factor
```

**Где:**
- early game: 0.3 - 0.6
- mid game: 0.6 - 0.9
- late game: 0.8 - 1.0

### 11.6. Ограничения систем

**Во время оффлайна:**

#### 11.6.1. Логистика

упрощается

считается мгновенной или усреднённой

#### 11.6.2. Производственные цепочки

только итоговый результат, без поцикловой симуляции

#### 11.6.3. Узкие места

bottlenecks учитываются через коэффициенты

### 11.7. Формула производства

```text
output = sum(all_buildings_production) * effective_time
```

### 11.8. Ограничение перепроизводства

output <= storage_limit

### 11.9. Переполнение склада

**Если превышение:**
- излишек теряетсяили
- конвертируется в бонус (опционально)

### 11.10. Влияние технологий

offline_bonus += tech_modifier

### 11.11. Влияние престижа

offline_efficiency += prestige_bonus

### 11.12. Состояние зданий

**При оффлайне:**
- ACTIVE → считается
- IDLE → 0
- BLOCKED → 0

### 11.13. Проблема "идеального оффлайна"

**Нельзя допускать:**

```text
offline = better than active play
```

**Решение:**
- штраф эффективности
- ограничение времени
- упрощение логики

### 11.14. UI требования

**Игрок должен видеть:**
- сколько времени прошло
- сколько ресурсов получено
- какие здания работали
- какие простаивали

### 11.15. Формула результата

```text
final_resources =production_rate* offline_time* efficiency_multiplier* caps
```

### 11.16. Edge cases

#### 11.16.1. Нулевое производство

ничего не начисляется

#### 11.16.2. Переполненный склад

лишнее теряется

#### 11.16.3. Отсутствие ресурсов

производство считается остановленным

### 11.17. Ошибки проектирования

Слишком точная симуляция (дорого)

Игнорирование bottlenecks

Отсутствие капа времени

Перекос в пользу оффлайна

### 11.18. Минимальный набор (definition of done)

**Система реализована, если:**
- есть расчёт оффлайн прогресса
- есть лимит времени
- есть упрощение симуляции
- есть награда игроку
- нет превышения эффективности активной игры

## 14. Эпохи (параллельные слои)

Эпохи (параллельные слои)

### 14.1. Определение

Эпоха — независимый производственный слой со своей экономикой, зданиями и ресурсами, работающий параллельно с другими эпохами.

### 14.2. Ключевая идея

эпохи существуют одновременно

игрок не "переходит", а "добавляет слой"

каждая эпоха остаётся активной до конца игры

### 14.3. Структура эпохи

```text
Epoch {id: stringname: stringresources: list<Resource>buildings: list<Building>technologies: list<Technology>is_unlocked: bool}
```

### 14.4. Примеры эпох

Каменный век

Античность

Индустриальная эпоха

Космическая эпоха

### 14.5. Поведение эпох

**Каждая эпоха:**
- имеет собственный production loop
- имеет собственные цепочки ресурсов
- работает независимо от других

### 14.6. Изоляция экономики

StoneAge.resources != Industrial.resources

**Но:**

могут взаимодействовать через связи эпох

### 14.7. Разблокировка эпох

unlock_epoch if:technology_requirements metAND global progression threshold reached

### 14.8. Добавление новой эпохи

**При открытии:**
- создаётся новый слой
- добавляются здания
- добавляются ресурсы
- включается независимая симуляция

### 14.9. Масштаб сложности

**Каждая новая эпоха:**
- увеличивает количество цепочек
- увеличивает требования
- увеличивает мультипликативные связи

### 14.10. Влияние на старые эпохи

new_epoch can modify old_epoch via bonuses

### 14.11. Прогресс внутри эпохи

**Каждая эпоха имеет:**
- собственное дерево технологий
- собственные здания
- собственные bottlenecks

### 14.12. Параллельная симуляция

**Все эпохи:**

run in parallel simulation ticks

### 14.13. Общий пул vs изоляция

**Внутри эпохи:**
- общий склад
- автоматическая логистика

**Между эпохами:**
- отдельные системы
- ограниченный обмен

### 14.14. Взаимодействие эпох

**Эпохи могут:**
- поставлять ресурсы друг другу
- усиливать производство
- открывать новые рецепты

### 14.15. Пример взаимодействия

Industrial steel -> Space rocket components

### 14.16. Ограничение взаимодействия

**Чтобы избежать хаоса:**
- кросс-эпохные связи ограничены
- есть коэффициенты ослабления
- cross_epoch_effect *= 0.3 - 0.7

### 14.17. UI требования

**Игрок должен видеть:**
- отдельные панели эпох
- текущий прогресс каждой
- ресурсы и производство
- связи между эпохами

### 14.18. Баланс

**Эпохи должны:**
- добавлять сложность
- не заменять старые системы
- создавать новые bottlenecks

### 14.19. Ошибки проектирования

Эпохи слишком независимы (нет смысла связывать)

Эпохи слишком связаны (теряется смысл слоёв)

Слишком быстрый рост сложности

Дублирование контента

## 15. Связь эпох (кросс-эффекты)

Связь эпох (кросс-эффекты)

### 15.1. Определение

Связь эпох — система, которая позволяет ресурсам, эффектам и бонусам из одной эпохи влиять на другие эпохи.

Это слой "межсистемной интеграции", который соединяет все производственные эпохи в единую экономику.

### 15.2. Типы связей

#### 15.2.1. Ресурсные связи

Epoch A -> Epoch B (resource transfer)

**Пример:**

дерево из каменного века используется в индустриальной эпохе

#### 15.2.2. Производственные бонусы

Epoch A production boosts Epoch B

**Пример:**

индустриальная автоматизация ускоряет добычу дерева

#### 15.2.3. Технологические связи

tech unlock in A affects B

#### 15.2.4. Глобальные эффекты

epoch contributes to global multiplier

### 15.3. Модель связи

```text
CrossEpochLink {from_epoch: Epochto_epoch: Epochtype: enumstrength: floatdecay: float}
```

### 15.4. Сила связи

```text
effective_strength = base_strength * modifiers
```

### 15.5. Ослабление связи

**Чтобы избежать доминирования одной эпохи:**

```text
final_effect = strength * 0.2 .. 0.7
```

### 15.6. Направленные связи

**Связи не обязательно двусторонние:**
- A -> B разрешено
- B -> A может отсутствовать

### 15.7. Обмен ресурсами

#### 15.7.1. Прямой

```text
transfer(resource, epoch_A, epoch_B)
```

#### 15.7.2. Через буфер

global_buffer -> distribution system

### 15.8. Производственные усиления

**Пример формулы:**

production_B *= (1 + output_A * coefficient)

### 15.9. Кросс-цепочки

**Позволяют строить:**
- цепочки между эпохами
- многоуровневые рецепты

### 15.10. Ограничения

#### 15.10.1. Лимит влияния

max_cross_epoch_bonus

#### 15.10.2. Убывающая отдача

```text
effectiveness = log(x) or sqrt(x)
```

### 15.11. Балансная модель

**Связи должны:**
- усиливать стратегию
- не заменять локальные цепочки
- создавать выбор

### 15.12. Риски системы

Одна эпоха становится "главной"

### 15.13. UI требования

**Игрок должен видеть:**
- граф связей между эпохами
- входящие и исходящие эффекты
- силу влияния
- bottleneck между эпохами

### 15.14. Минимальный набор (definition of done)

**Система считается реализованной, если:**
- есть хотя бы 2 типа кросс-связей
- ресурсы могут пересекать эпохи
- есть влияние одной эпохи на другую
- есть ограничения силы эффектов
- игрок может это понимать через UI
